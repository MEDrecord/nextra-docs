# B2C Gateway Integration - Complete Reference Documentation

## Project Overview

MEDrecord B2C Gateway Integration is a comprehensive enterprise authentication and API gateway system built on Next.js that provides:

- Multi-tenant Azure B2C integration with automatic synchronization
- Role-based access control (RBAC) with both global and tenant-specific role management
- API gateway with multiple authentication schemes and service proxying
- Mobile authentication with dedicated token exchange and session management
- Cross-domain authentication via web token exchange system
- Advanced security features including CSRF protection, rate limiting, IP restrictions, and audit logging
- Feature flag system for progressive rollout and A/B testing

## Authentication System

### 1. Web Authentication Flow

Web authentication handles browser-based user login via Azure B2C with PKCE authorization flow.

**Flow Diagram:**

```
1. User visits app
   ↓
2. Redirect to Azure B2C signin
   ↓
3. User authenticates with MFA (if required)
   ↓
4. Azure B2C redirects to /auth/callback with auth code
   ↓
5. Backend exchanges code for tokens (confidential)
   ↓
6. Create HttpOnly session cookie
   ↓
7. Redirect to dashboard
```

**Key Files:**
- `/lib/auth/b2c-flows.ts` - PKCE flow implementation
- `/lib/auth/session.ts` - Session cookie management
- `/app/api/auth/signin/route.ts` - Sign-in endpoint
- `/app/api/auth/callback/route.ts` - OAuth callback handler

**Session Cookie Structure:**

```javascript
// HttpOnly, Secure, SameSite=Strict
{
  sessionId: string,           // Unique session identifier
  userId: string,              // Azure B2C object ID
  email: string,               // User email
  tenantId: string,            // Current tenant context
  roles: string[],             // User roles in current tenant
  globalRoles: string[],       // System-wide roles
  lastActivity: timestamp,     // Last interaction time
  createdAt: timestamp,        // Session creation time
  mfaVerified: boolean,        // MFA status
  nonce: string                // CSRF protection nonce
}
```

### 2. Mobile Authentication Flow

Mobile apps use a dedicated login code system optimized for native app integration.

**Flow Diagram:**

```
1. Mobile app requests login code
   ↓
2. Backend generates 6-digit code + session
   ↓
3. User enters code in browser (or scans QR)
   ↓
4. Browser authenticates with Azure B2C
   ↓
5. Server links browser session to mobile session
   ↓
6. Mobile exchanges code for mobile JWT token
   ↓
7. Mobile app stores token securely
```

**Endpoints:**
- `POST /api/auth/mobile/login-code` - Request login code
- `POST /api/auth/mobile/exchange` - Exchange code for token
- `POST /api/auth/mobile/logout` - Logout and revoke token

**Mobile Token Format (JWT):**

```javascript
{
  sub: string,                 // User ID (Azure B2C object ID)
  email: string,
  tenantId: string,
  roles: string[],
  mobileSessionId: string,     // Unique mobile session ID
  deviceId: string,            // Device fingerprint/identifier
  issuer: "B2C-Gateway",
  iat: timestamp,              // Issued at
  exp: timestamp,              // Expires (24 hours)
  nonce: string                // Replay attack prevention
}
```

**Key Files:**
- `/lib/auth/mobile-sessions.ts` - Mobile session management
- `/lib/auth/mobile-pop.ts` - Proof-of-possession for mobile
- `/app/api/auth/mobile/exchange/route.ts` - Token exchange

### 3. Web Token Exchange (Cross-Domain Authentication)

For scenarios requiring authentication transfer between domains (e.g., B2C web app → microservice portal), one-time use web tokens enable secure cross-domain authentication.

**Flow:**

```
1. Authenticated web user requests one-time token
   ↓
2. Backend generates token (60-second TTL)
   ↓
3. User/app redirects to target domain with token as query param
   ↓
4. Target domain validates token (one-time use only)
   ↓
5. Target domain creates its own session
   ↓
6. Token invalidated (cannot be reused)
```

**Endpoints:**
- `POST /api/auth/web-session/exchange` - Generate one-time token
- `POST /api/auth/web-session/verify` - Verify token and create session

**Use Cases:**
- Single Sign-On (SSO) across internal applications
- Secure handoff to partner systems
- Portal login redirects

**Key Files:**
- `/lib/auth/web-tokens.ts` - Web token generation and validation
- `/app/api/auth/web-session/exchange/route.ts` - Token generation
- `/app/api/auth/web-session/verify/route.ts` - Token validation

### 4. Azure B2C Synchronization

The system automatically synchronizes user and tenant data from Azure B2C to maintain consistent state.

**Sync Features:**
- Automatic user sync when logging in
- Tenant provisioning on first auth
- Role mapping from B2C custom attributes
- Graph API integration for admin operations
- Retry logic with exponential backoff

**Key Files:**
- `/lib/auth/b2c-tenant-sync.ts` - Tenant/user synchronization
- `/lib/auth/b2c-graph.ts` - Microsoft Graph API integration
- `/app/api/cron/process-b2c-sync-queue/route.ts` - Async sync processor

## Login Pages & User Interface

This section documents the Gateway-hosted authentication UI pages. These pages are part of the Gateway itself and are used by all consuming applications (web apps, mobile apps, external integrations) through redirect flows.

### How External Apps Use These Pages

External applications do NOT create their own login UI. Instead, they redirect users to the Gateway's login pages:

```
 Your App (any domain)                    This Gateway                    Azure B2C
┌────────────────────┐              ┌────────────────────┐              ┌──────────┐
│                    │              │                    │              │          │
│ 1. User clicks     │              │                    │              │          │
│    "Sign In"       │──────────────│ 2. /auth/signin    │──────────────│ 3. B2C   │
│                    │   redirect   │    page loads      │   redirect   │    login │
│                    │              │                    │              │          │
│ 6. Receive token   │◄─────────────│ 5. Process B2C     │◄─────────────│ 4. User  │
│    or sessionId    │   redirect   │    callback        │   callback   │    auth  │
│                    │   + token    │                    │              │          │
└────────────────────┘              └────────────────────┘              └──────────┘
```

**Integration patterns:**

| App Type | Integration Method | What the App Receives |
|----------|-------------------|----------------------|
| Same-domain web app | Redirect to `/api/auth/signin` | Session cookie (automatic) |
| Cross-domain web app | Redirect to `/api/auth/signin?callbackUrl=...` | webToken in URL, exchange for sessionId |
| Mobile app (React Native) | System browser to `/api/auth/signin?clientNonce=...` | loginCode in callback URL, exchange for sessionToken |
| Machine-to-machine | No login UI needed | Uses API keys (`cak_`, `sak_`) |

| `/auth/error` | Error Page | Authentication error display with i18n |

### 1. Sign In Page (`/auth/signin`)

The sign-in page is the primary entry point for user authentication. All external applications redirect here - they do not implement their own login forms.

**Query Parameters (for external apps):**

| Parameter | Required | Description | Example |
|-----------|----------|-------------|---------|
| `callbackUrl` | Yes (for cross-domain) | URL to redirect after auth | `https://myapp.vercel.app/auth/callback` |
| `tenantId` | Optional | Tenant context for the session | `tenant_abc123` |
| `clientNonce` | Mobile only | Binds login to mobile app instance | Random 32+ byte string |

**How to redirect here from your app:**

```javascript
// Same-domain web app (cookie-based)
window.location.href = '/api/auth/signin?callbackUrl=/dashboard';

// Cross-domain web app (webToken exchange)
const callback = encodeURIComponent('https://myapp.vercel.app/auth/callback');
window.location.href = `https://gateway.example.com/api/auth/signin?callbackUrl=${callback}`;

// Mobile app (loginCode exchange)
const callback = encodeURIComponent('healthtalk://auth/callback');
const nonce = generateClientNonce();
InAppBrowser.openAuth(`https://gateway.example.com/api/auth/signin?callbackUrl=${callback}&clientNonce=${nonce}`);
```

## Authorization (RBAC)

### Role Hierarchy

The system supports two levels of role management:

### 1. Global Roles

System-wide roles that apply across all tenants:

| Role | Permission | Use Case |
|------|------------|----------|
| Super Admin | All system operations | System maintenance |
| Global Admin | Tenant management, user management | Global administration |
| Developer | Service creation, gateway key management | API integration |
| Operator | Analytics, audit logs, monitoring | Operations team |

### 2. Tenant-Specific Roles

Each tenant can define custom roles for fine-grained access control.

**Built-in Tenant Roles:**
- `TenantAdmin` - Full tenant management
- `Developer` - Service and API key management
- `User` - Basic data access
- Custom roles - User-defined with specific permissions

**Permission Model:**

```javascript
{
  roleId: string,
  tenantId: string,
  name: string,                     // e.g., "Data Analyst"
  description: string,
  permissions: string[],            // e.g., ["read:analytics", "export:data"]
  createdAt: timestamp,
  createdBy: string,
  isSystem: boolean                 // True for built-in roles
}
```

### Permission System

Permissions follow a `resource:action` pattern:

**Common Permissions:**
- `read:dashboard` - View dashboard
- `write:settings` - Modify settings
- `manage:users` - Add/remove users
- `manage:roles` - Create/edit roles
- `create:service` - Create API service
- `manage:keys` - Generate API keys
- `read:analytics` - View analytics
- `read:audit` - View audit logs
- `export:data` - Export data

### Authorization Checks

**Path-Based RBAC:**
Protected routes enforce role requirements:

```javascript
// app/dashboard/admin/users
// Requires: Global role "Global Admin"

// app/dashboard/tenants/[tenantId]/settings
// Requires: Tenant role "TenantAdmin" in specified tenant
```

**API Endpoint RBAC:**
Backend routes validate permissions via middleware:

```javascript
// POST /api/tenant/[tenantId]/users
// Required permission: "manage:users"
// Scope: Current tenant
```

**Key Files:**
- `/lib/rbac/types.ts` - Role and permission type definitions
- `/lib/rbac/tenant-middleware.ts` - Tenant-specific RBAC enforcement
- `/lib/rbac/middleware.ts` - Global RBAC middleware
- `/lib/rbac/tenant-permissions.ts` - Permission definitions
- `/app/api/tenant/[tenantId]/roles/route.ts` - Role management API

## API Gateway

### Overview

The API gateway provides a unified entry point for backend service authentication, request routing, and security enforcement.

### Gateway Key System

Backend services authenticate via HMAC-signed requests using gateway keys.

**Key Format:**

```
gwk_<prefix>_<random-bytes>_<random-bytes>

Example: gwk_svc_abc123def456ghi789jkl012mno345
```

**HMAC Signing:**

```
Signature = HMAC-SHA256(
  message = HTTP_METHOD + '\n' + PATH + '\n' + BODY,
  key = gateway_key_secret
)
```

**Request Header:**

```
X-Gateway-Signature: <signature>
X-Gateway-Key-ID: gwk_svc_abc123def456ghi789jkl012mno345
```

**Key Rotation:**
- Keys have optional TTL configuration
- Regeneration creates new key + grace period for old key
- Automatic key retirement after TTL expires

### Service Registration

Backend services must be registered to use the gateway.

**Service Configuration:**

```javascript
{
  id: string,                       // Service UUID
  name: string,                     // Display name
  slug: string,                     // URL-safe identifier
  baseUrl: string,                  // Backend service URL
  description: string,
  gatewayKey: {
    id: string,
    expiresAt?: timestamp,
    rotatedAt: timestamp
  },
  endpoints: {
    path: string,
    methods: string[],
    publicAccessibleKey?: string,   // For public endpoints
    requiresAuth: boolean
  }[],
  rateLimitConfig: {
    requestsPerMinute: number
  },
  tenantId?: string,                // Tenant association (if multi-tenant service)
  createdAt: timestamp,
  createdBy: string
}
```

**Management Endpoints:**
- `GET /api/gateway/services` - List services
- `POST /api/gateway/services` - Create service
- `GET /api/gateway/services/[serviceId]` - Get service details
- `PUT /api/gateway/services/[serviceId]` - Update service
- `DELETE /api/gateway/services/[serviceId]` - Delete service
- `GET /api/gateway/services/[serviceId]/openapi` - Get OpenAPI spec

### Request Proxying

**Request Flow:**

```
1. Client makes request to /api/gateway/proxy/[...path]
   ↓
2. Gateway authenticates client (session or API key)
   ↓
3. Gateway validates authorization based on service permissions
   ↓
4. Gateway checks rate limiting
   ↓
5. Gateway logs request (audit)
   ↓
6. Gateway HMAC-signs request with service gateway key
   ↓
7. Forward to backend service
   ↓
8. Log response (audit)
   ↓
9. Return response to client
```

**Proxy Endpoint:**

```
POST /api/gateway/proxy/[service-slug]/[endpoint-path]
POST /api/gateway/proxy/users/list
POST /api/gateway/proxy/crm/contacts/search
```

**Request Headers Forwarded:**

```
X-Forwarded-For: <original-ip>
X-Forwarded-Proto: https
X-Forwarded-Host: <gateway-host>
X-User-ID: <current-user>
X-Tenant-ID: <current-tenant>
X-Gateway-Signature: <hmac-signature>
X-Gateway-Key-ID: <service-key-id>
```

**Key Files:**
- `/lib/gateway/proxy.ts` - Request proxying logic
- `/lib/gateway/crypto.ts` - HMAC signing
- `/app/api/gateway/proxy/[...path]/route.ts` - Proxy route handler
- `/lib/gateway/service-registry.ts` - Service registration management

## Security Features

### 1. CSRF Protection

All state-modifying requests include CSRF tokens.

**Implementation:**
- Tokens generated per-session with 1-hour TTL
- Auto-refresh when approaching expiration
- Validated on all POST/PUT/DELETE requests
- `SameSite=Strict` cookie policy

**Endpoints:**
- `POST /api/auth/csrf` - Get current CSRF token

**Key Files:**
- `/lib/security/csrf.ts` - CSRF token generation/validation
- `/lib/api-gateway/middleware.ts` - CSRF middleware

### 2. Rate Limiting

Multi-level rate limiting protects against abuse:

**Levels:**
1. Global rate limit - Per-IP across entire system
2. Per-user rate limit - Per authenticated user
3. Per-API-key rate limit - Per service API key
4. Per-endpoint rate limit - Per service endpoint

**Configuration:**

```javascript
{
  globalLimit: 1000,              // Requests per minute per IP
  perUserLimit: 100,              // Requests per minute per user
  perKeyLimit: 500,               // Requests per minute per API key
  perEndpointLimit: 10000         // Requests per minute per endpoint
}
```

**Rate Limit Headers:**

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1643723400
```

**Implementation:**
- Redis-backed counters for distributed rate limiting
- Token bucket algorithm
- Sliding window expiration

**Key Files:**
- `/lib/api-gateway/rate-limiter.ts` - Rate limiting logic
- `/lib/auth/session-security.ts` - Session rate limiting

### 3. IP Restrictions

Services can restrict access to specific IP ranges.

**Configuration:**

```javascript
{
  serviceId: string,
  whitelist: ["203.0.113.0/24", "203.0.113.42"],  // CIDR notation
  blacklist: ["192.0.2.0/24"],
  enforceClientIP: boolean        // Respect X-Forwarded-For
}
```

**Validation:**
- Checked on every gateway request
- Supports CIDR notation
- Both whitelist and blacklist modes
- IPv4 and IPv6 support

### 4. Request Validation

All incoming requests are validated for security and correctness.

**Validation Steps:**
1. Content-Type checking - Validates media type
2. Request size limits - Prevents payload attacks
3. JSON schema validation - Validates request structure
4. Input sanitization - Escapes/removes dangerous content
5. SQL injection prevention - Parameterized queries only

**Configuration:**

```javascript
{
  maxRequestSize: 10 * 1024 * 1024,  // 10 MB
  maxJsonSize: 1 * 1024 * 1024,      // 1 MB
```

## Multi-Tenant System

### Tenant Architecture

Each tenant is an isolated business unit with:
- Independent user base
- Tenant-specific roles and permissions
- Dedicated API services and keys
- Separate analytics and audit logs
- Custom configuration

### Tenant Context

User operations are scoped to their current tenant:

```javascript
// User can be in multiple tenants
{
  userId: string,
  tenants: [
    {
      tenantId: string,
      name: string,
      role: string,               // User's role in this tenant
      isDefault: boolean
    }
  ],
  currentTenantId: string        // Active context
}
```

### Tenant Switching

Users can switch between tenants:

**Endpoints:**
- `POST /api/tenant/switch` - Switch active tenant
- `GET /api/user/tenants` - List user's tenants

**Session Update:**
When switching tenants, the session is updated with:
- New tenantId
- New role set for that tenant
- New permission set
- Updated audit context

### Tenant-Scoped Operations

Most operations are automatically scoped to the current tenant:

```
GET /api/tenant/[tenantId]/users - List users in tenant
POST /api/tenant/[tenantId]/users - Add user to tenant
GET /api/tenant/[tenantId]/roles - List tenant roles
POST /api/tenant/[tenantId]/roles - Create tenant role
GET /api/tenant/[tenantId]/settings - Get tenant settings
```

### Tenant Isolation

**Data Isolation:**
- Queries filtered by tenantId
- Row-level security at database level
- No cross-tenant data leakage

**API Key Isolation:**
- Service API keys scoped per tenant
- Client API keys associated with specific tenants
- Gateway keys cannot access data outside scoped tenant

**Audit Isolation:**
- Audit logs filtered by tenant
- Tenant admins only see their own tenant logs

**Key Files:**
- `/lib/tenant/context.ts` - Tenant context management
- `/lib/tenant/resolver.ts` - Tenant context resolution
- `/app/api/tenant/[tenantId]/` - Tenant-scoped endpoints

## Mobile Integration

### Mobile Authentication Overview

Mobile apps have specialized authentication flow optimized for native environments:

**Advantages:**
- 6-digit login code (easy to use)
- No password entry needed in mobile app
- Integrates with device security (biometric, PIN)
- Automatic device fingerprinting
- Secure token storage recommendations

### Mobile Session Token

Mobile sessions are managed with JWT tokens instead of cookies:

**Token Claims:**

```javascript
{
  sub: string,                 // User ID
  email: string,
  tenantId: string,
  roles: string[],
  deviceId: string,            // Device fingerprint
  mobileSessionId: string,     // Session identifier
  nonce: string,               // Replay prevention
  iat: number,                 // Issued at
  exp: number,                 // Expires (24 hours)
  iss: string                  // Issuer
}
```

**Key Rotation:**
- Tokens expire after 24 hours
- Refresh tokens rotate on each use
- Grace period for token rotation (5 minutes)
- Server tracks all active mobile sessions per user

### Mobile Authentication Endpoints

**1. Request Login Code**

```
POST /api/auth/mobile/login-code

Request:
{
  deviceId: string,            // Device fingerprint
  appIdentifier: string        // App package/bundle ID
}

Response (201):
{
  loginCode: "123456",
  expiresIn: 600,              // Seconds
  deviceToken: string          // For proof-of-possession
}
```

**2. Exchange Code for Token**

```
POST /api/auth/mobile/exchange

Request:
{
  loginCode: "123456",
  deviceToken: string,
  proof: string                // HMAC proof of possession
}

Response (200):
{
  accessToken: string,         // JWT token
  refreshToken: string,
  expiresIn: 86400,            // 24 hours
  tokenType: "Bearer"
}
```

**3. Logout/Revoke**

```
POST /api/auth/mobile/logout

Headers:
Authorization: Bearer <accessToken>

Response (200):
{
  success: true
}
```

### Mobile Session Management

**Proof-of-Possession (PoP):**
Prevents token theft if device is compromised.

```javascript
// Generate PoP proof
proof = HMAC-SHA256(
  message = deviceToken + timestamp + nonce,
  key = deviceId
)
```

**Device Fingerprinting:**
- Collects device identifiers (anonymously)
- Used to detect token theft
- Can enforce single-device mode

### Mobile Integration Best Practices

**Security Recommendations:**
1. Store tokens in secure enclave/keychain
2. Implement certificate pinning
3. Validate SSL certificates
4. Never log tokens
5. Clear tokens on logout
6. Implement biometric unlock on app launch

**Implementation Guide:**
See `/docs/MOBILE_GATEWAY_INTEGRATION.md` for platform-specific examples.

**Key Files:**
- `/lib/auth/mobile-sessions.ts` - Mobile session management
- `/lib/auth/mobile-pop.ts` - Proof-of-possession
- `/app/api/auth/mobile/exchange/route.ts` - Token exchange
- `/app/api/auth/mobile/logout/route.ts` - Logout

## Troubleshooting

### Common Issues

**Session not persisting:**
- Check `SESSION_SECRET` is set
- Verify cookie domain matches hostname
- Check browser allows third-party cookies (if applicable)

**Mobile token expiration:**
- Mobile tokens expire after 24 hours
- Implement refresh token flow
- Check server time sync

**Rate limiting too strict:**
- Adjust `RATE_LIMIT_*` environment variables
- Check per-key limits not too low
- Monitor rate limit headers in responses

**CSRF token errors:**
- Ensure CSRF endpoint called before state-modifying requests
- Check CSRF token included in `X-CSRF-Token` header
- Token auto-refreshes within 1 hour

**Multi-tenant user access issues:**
- Verify user added to tenant via `/api/tenant/[tenantId]/users`
- Check role assigned in target tenant
- Try switching tenant via `/api/user/switch-tenant`
