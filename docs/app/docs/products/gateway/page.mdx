# Gateway

The MEDrecord Gateway is a secure API Gateway that provides unified authentication, routing, and request validation for all MEDrecord products and web applications.

## Overview

Gateway serves as the central entry point for all authentication and API requests, ensuring consistent security, session management, and audit logging across the entire product ecosystem.

### Key Features

- **Azure B2C Integration** - Single sign-on via Azure Active Directory B2C
- **Session Cookie Management** - HttpOnly secure cookies for web applications
- **HMAC Authentication** - Cryptographic request signing for backend services
- **Multi-Tenant Routing** - Automatic tenant context injection
- **Role-Based Access Control** - Path-based access restrictions per role
- **Audit Logging** - Comprehensive request/response logging

## Service Types

Gateway supports two types of services:

| Type | Authentication | Example |
|------|----------------|---------|
| **Web App** | Session cookie (from Azure B2C) | Docs, Helpdesk, HealthTalk |
| **Backend Service** | HMAC signature | EHR API, AI Services |

### Web App Services

Web applications authenticate users through Azure B2C and receive session cookies from Gateway:

```
┌──────────────┐     ┌─────────────────────────────────┐
│   Browser    │     │          API Gateway            │
│              │     │                                 │
└──────┬───────┘     └────────────┬────────────────────┘
       │                          │
       │  1. Sign In              │
       ├─────────────────────────>│ → Azure B2C
       │                          │
       │  2. Session Cookie       │
       │<─────────────────────────┤
       │     (auth.sid)           │
       │                          │
       │  3. Access Web App       │
       │     (Cookie included)    │
       ├─────────────────────────>│
       │                          │
       │  4. Validate Cookie      │
       │     Return content       │
       │<─────────────────────────┤
```

**Example: Docs Service**

The Docs service (`docs-tst.healthtalk.ai`) is a web application that:

1. User authenticates via Helpdesk or any MEDrecord app
2. Gateway issues `auth.sid` session cookie
3. User navigates to Docs - cookie is automatically included
4. Docs middleware validates cookie with Gateway
5. Gateway returns user info including role
6. Docs applies path-based access control based on role

### Backend Services

Backend services use HMAC signatures for secure server-to-server communication:

```
┌──────────────┐     ┌─────────────────────────────────┐     ┌──────────────┐
│   Frontend   │     │          API Gateway            │     │   Backend    │
└──────┬───────┘     └────────────┬────────────────────┘     └──────┬───────┘
       │                          │                                  │
       │  API Call (Cookie)       │                                  │
       ├─────────────────────────>│                                  │
       │                          │  Sign Request (HMAC)             │
       │                          ├─────────────────────────────────>│
       │                          │  X-Gateway-Key: gwk_xxx          │
       │                          │  X-Gateway-Signature: ...        │
       │                          │  X-User-Id: user123              │
       │                          │                                  │
       │                          │  Verify & Process         <──────┤
       │  Response                │                                  │
       │<─────────────────────────┤<─────────────────────────────────┤
```

## Cookie-Based Authentication

### Session Cookie

After Azure B2C sign-in, Gateway issues:

```
Set-Cookie: auth.sid=<session_token>; 
  HttpOnly; 
  Secure; 
  SameSite=Lax; 
  Domain=.healthtalk.ai
```

**Properties:**
- `HttpOnly` - Not accessible via JavaScript (XSS protection)
- `Secure` - Only sent over HTTPS
- `SameSite=Lax` - CSRF protection
- `Domain=.healthtalk.ai` - Shared across all subdomains

### Cross-App Authentication

Because the cookie domain is `.healthtalk.ai`, users authenticated to one app can access other apps:

| App | URL | Access |
|-----|-----|--------|
| Helpdesk | helpdesk-tst.healthtalk.ai | Cookie shared |
| Docs | docs-tst.healthtalk.ai | Cookie shared |
| HealthTalk | healthtalk-tst.healthtalk.ai | Cookie shared |

### Validating Cookies

Web apps validate cookies by calling Gateway:

```javascript
const response = await fetch(`${GATEWAY_URL}/api/user/me`, {
  headers: {
    'Cookie': `auth.sid=${cookieValue}`,
  },
  cache: 'no-store',
})

const { user } = await response.json()
// user = { id, email, role, tenantId, ... }
```

## Role-Based Access Control (RBAC)

### Path-Based Restrictions

Web apps can restrict paths based on user roles:

```typescript
// Example: Docs middleware configuration
const PATH_ROLES = {
  '/isms/policies': ['user', 'admin', 'security_officer'],
  '/isms/risks': ['admin', 'security_officer'],
  '/isms/audit': ['admin', 'security_officer'],
  '/isms/registers': ['admin'],
  '/admin': ['admin'],
}
```

### Role Hierarchy

| Role | Description | Access Level |
|------|-------------|--------------|
| `user` | Standard authenticated user | Public paths only |
| `reporter` | Reporting access | + Report paths |
| `security_officer` | Security management | + ISMS restricted |
| `admin` | Full administrative access | All paths |

## HMAC Authentication (Backend)

For backend services, Gateway uses HMAC signing:

### Request Headers

```
X-Gateway-Key: gwk_xxx (identifies the Gateway)
X-Gateway-Signature: HMAC-SHA256 signature
X-User-Id: Authenticated user identifier
X-Tenant-Id: Tenant/organization identifier
X-Tenant-Role: User role within tenant
X-Request-Id: Unique request identifier
```

### Signature Calculation

```javascript
const signature = crypto
  .createHmac('sha256', serviceSecretKey)
  .update(JSON.stringify(requestBody))
  .digest('hex')
```

## Key System

| Key Type | Purpose | Used By |
|----------|---------|---------|
| `gwk_` | Gateway Key - Identifies Gateway to Backend | Gateway → Backend |
| `ssk_` | Service Secret Key - HMAC signing (never transmitted) | Both sides |
| `sak_` | Service API Key - Admin operations | Admin tools |
| `cak_` | Client API Key - Machine-to-machine | External integrations |

**Important:** Frontend web apps use session cookies only - no API keys needed!

## Security

### Request Validation

- JSON schema validation

### Audit Logging

Every request is logged with:

- Timestamp
- User and tenant context

Logs are immutable and retained per compliance requirements.

## Integration

### For Web Apps

```typescript
// middleware.ts
async function validateSession(request: NextRequest) {
  const authCookie = request.cookies.get('auth.sid')
  if (!authCookie?.value) return null

  const response = await fetch(`${GATEWAY_URL}/api/user/me`, {
    headers: { 'Cookie': `auth.sid=${authCookie.value}` },
  })

  if (!response.ok) return null
  return (await response.json()).user
}
```

### For Backend Services

```typescript
// Verify incoming Gateway request
function verifyGatewaySignature(req, secretKey) {
  const signature = req.headers['x-gateway-signature']
  const expected = crypto
    .createHmac('sha256', secretKey)
    .update(JSON.stringify(req.body))
    .digest('hex')
  return signature === expected
}
```
